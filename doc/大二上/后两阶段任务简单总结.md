# 报告一：Jetson Nano 底层驱动集成与 ROS 2 动力学实现

概述：本任务在 Jetson Nano 上成功部署了 ROS 2 Humble 环境，通过编写核心驱动节点实现了麦克纳姆轮的逆运动学解算，实现了通过ROS2系统在Jetson Nano 本地控制小车运动。

## 一、 实现过程 (Implementation Process)

### 1.1 硬件层：物理链路与驱动适配
首先完成 Jetson Nano 与底层电机驱动板的物理连接。通过 I2C/串口协议，在 Ubuntu 系统中通过 `ls /dev/tty*` 确认通讯端口，并赋予用户读写权限。整合厂家提供的底层 SDK，确保编码器反馈与电机控制 PWM 信号能够被 ROS 2 节点正确读取。

### 1.2 环境层：ROS 2 Humble 系统部署
在嵌入式端安装 ROS 2 Humble，配置 Workspace（工作空间）。关键步骤包括设置 `COLCON` 构建系统，并编译 `yahboomcar_bringup` 与 `geometry_msgs` 等核心包，建立起“算法-消息-执行”的闭环环境。

### 1.3 逻辑层：麦克纳姆轮动力学解算
实现从线性速度到单轮转速的逆运动学转换。根据麦克纳姆轮物理特性，建立如下数学模型：
$$V_{wheel\_FL} = V_x - V_y - \Omega_z(L+W)$$
$$V_{wheel\_FR} = V_x + V_y + \Omega_z(L+W)$$
其中 $L$ 和 $W$ 为底盘轴距与轮距参数。代码逻辑将接收到的 `/cmd_vel` 话题数据实时解算为四路 PWM 占空比。



## 二、 关键问题解决 (Key Problems Solved)

### 2.1 ROS 2 动态环境加载与 Python 路径冲突
* **【问题描述】**：在 Jetson Nano 终端或后台脚本调用 `rclpy` 模块时，系统常报错找不到模块，或因 Python 虚拟环境与系统 ROS 环境不兼容导致驱动加载失败。
* **【解决方案】**：编写了专门的启动逻辑：在 Python 脚本入口通过 `sys.path.append` 注入 ROS 系统库路径，并配套 `.sh` 自动化脚本，在执行驱动前强制执行 `source /opt/ros/humble/setup.bash`。这一机制保证了底层通讯中间件（DDS）在嵌入式端环境中的稳定性。

### 2.2 电机低速“死区”与平移偏航补偿
* **【问题描述】**：实验发现，当给定的平移速度 $V_y$ 较低时，由于电机静摩擦力和驱动电压不均，小车无法启动或出现严重的斜向偏航。
* **【解决方案】**：
    * **死区补偿**：在代码中设置最小 PWM 输出阈值（Dead-zone），确保一旦接收到指令，电机电流能瞬间突破静摩擦力。
    * **增益校准**：通过多次实测，为 $V_y$ 分量引入了 1.15 倍的增益补偿系数，解决了因麦轮滚子侧向分力损失导致的动作迟滞问题。

## 三、 阶段性成果
成功实现了本地键盘对底盘的 8 方向全向控制，话题通讯频率稳定在 30Hz，指令延迟低于 35ms，为远程可视化系统的开发提供了稳健的物理载体。















# 报告二：基于 TCP 协议与 PyQt6 的机器人远程控制系统实现

概述：本项目利用 PyQt6 构建可视化界面，通过 TCP 协议与 Jetson Nano 建立异步通讯链路，实现了将 PC 端操控动作实时转化为 ROS 2 指令，并结合底层事件拦截逻辑彻底解决按键冲突，打造了一个跨平台、高响应的全向机器人远程监控终端。

## 一、 实现过程 (Implementation Process)

### 1.1 通讯层：TCP 双端链路构建
系统采用 **Client-Server 架构**。在 Jetson Nano 端运行 TCP 服务端（Server），利用多线程处理 Socket 监听；在 PC 控制端运行客户端（Client），建立长连接链路。该方案绕过了 ROS 2 在跨网段通讯时的复杂配置，实现了更轻量化的异构设备通讯。

### 1.2 表现层：基于 PyQt6 的工业级 UI 设计
使用 PyQt6 开发桌面级控制软件。通过 `QGridLayout` 网格布局设计 3×3 的控制矩阵。
* **视觉反馈**：通过 QSS 样式表配置“暗色系”界面，并为每个按钮配置 `:pressed` 伪状态实现交互反馈。
* **双重映射**：同时绑定 `QPushButton.clicked` 事件与主窗口的键盘捕获事件。

### 1.3 指令网关：语义化指令转发逻辑
在机器人端开发“翻译”节点，将收到的字符串（如 `forward`, `left`, `stop`）在毫秒级内通过 Python 字典映射为对应的 `Twist` 消息分量，并实时发布至 `/cmd_vel` 话题。



## 二、 关键问题解决 (Key Problems Solved)

### 2.1 UI 键盘焦点冲突与“空格键失效”难题
* **【问题描述】**：在使用键盘 WASD 控制过程中，一旦鼠标点击了某个按钮（如“前进”），该按钮会占据焦点。此时按下“空格键（Space）”刹车时，系统会将其默认解释为“点击当前按钮”，导致小车不仅不停止，反而再次执行前进，存在极大安全隐患。
* **【解决方案】**：
    * **焦点剥离策略**：对所有 UI 按钮执行 `setFocusPolicy(Qt.FocusPolicy.NoFocus)`，剥夺按钮对键盘信号的拦截权。
    * **全局事件过滤器（Event Filter）**：在系统底层安装 Hook 钩子。一旦检测到物理按键 `Qt.Key_Space`，立即强行中断所有子组件的默认逻辑分发，优先执行 `send_cmd('stop')`。该方案实现了 100% 的刹车响应优先级。

### 2.2 TCP 链路粘包处理与异常断连保护
* **【问题描述】**：高频操作下，多条控制指令可能在缓冲区粘连。若 PC 端程序闪退或网络波动，小车会因为收不到“停止”信号而继续保持失控行驶。
* **【解决方案】**：
    * **指令清洗**：接收端引入 `strip()` 与 `split()` 预处理，确保每条报文独立解析，仅执行最新的动作指令。
    * **心跳检测与安全刹车**：逻辑网关通过捕获 `socket.error` 实现连接感知。一旦网络断开，程序自动触发紧急保护回调，向底盘下发全 0 的速度向量，强制小车即刻进入“安全静止”状态。

## 三、 阶段性成果
构建了一套响应迅速、逻辑严密的远程监控系统。通过事件拦截机制彻底解决了操控冲突问题，实现了 150 米范围内的低延迟稳定控制（WiFi 环境下），具备工业级操控终端的初步形态。
